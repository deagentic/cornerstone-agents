# AGENTS.server+frontend.md — Server + Frontend Integration Routing

## Purpose

This fragment is injected automatically when both the `server` and `frontend` capabilities are present in a project. It governs the Backend-for-Frontend (BFF) integration patterns.

## Activate automatically when

Both `capabilities/server/` and `capabilities/frontend/` are present in the sparse checkout.

## Integration Concerns

### Authentication

- The server is the single source of truth for auth (JWT issuance, session management).
- Frontend receives tokens via secure HTTP-only cookies or `Authorization` headers — never `localStorage`.
- CORS configuration on the FastAPI server must explicitly allowlist the frontend origin.

### API Contract Design

- Use OpenAPI schema (auto-generated by FastAPI) as the contract boundary.
- Frontend must consume the OpenAPI spec to generate typed API clients — do not hand-write fetch calls.
- Breaking API changes require a version bump (`/v2/`) and a migration plan communicated to frontend.

### BFF Pattern

For complex frontend pages that aggregate multiple domain entities, create a BFF route in the server:

```
frontend page → BFF route (server/adapters/inbound/bff/) → multiple domain use cases → composed response
```

This avoids N+1 round trips from the frontend.

### Routing Rules

- When designing a new page that requires data from the server, first define the BFF API shape with `architect` (from `common/`), then implement both sides.
- When auth behavior changes on the server, always update the frontend auth flows in the same PR.
- CORS and auth configuration changes require review from `security-expert` (from `common/`).
